# YouTube Summary Telegram Bot

Личный Telegram-бот: принимает ссылку на YouTube (или подписывает канал на
мониторинг), получает transcript или транскрибирует аудио через Groq Whisper
API, генерирует summary через LLM (локальная модель в LM Studio или
OpenRouter) и отправляет краткую версию в Telegram, а полный конспект
публикует в Telegra.ph.

## Что уже настроено

- Telegram long polling через `aiogram`.
- Доступ ограничен allow-list'ом: `OWNER_USER_ID` для владельца,
  `ALLOWED_USER_IDS` как seed при первом запуске, дальше список пользователей
  живёт в SQLite.
- Summary — через LM Studio (локально) или OpenRouter, с circuit breaker'ом
  на случай простоя провайдера (см. `/llm_mode`, `/llm_paid`).
- Транскрибация роликов без субтитров через Groq Whisper API.
- `yt-dlp` используется как fallback для скачивания аудио и может использовать
  YouTube cookies.
- Все состояние (пользователи, кэш саммари, очередь задач, дайджесты,
  состояние мониторинга, бюджет OpenRouter) хранится в единой SQLite-базе
  `data/bot.db`.
- Очередь генерации саммари персистентна: незавершённые задачи переживают
  рестарт бота.
- Мониторинг YouTube-каналов с одним утренним LLM-ранжированным дайджестом
  вместо потока отдельных сообщений.
- Browser-extension (Chrome/Firefox) с кнопкой на странице YouTube-видео —
  открывает бота deep-link'ом на конкретный ролик.
- Мультиязычность на 7 языков (`ru`/`en`/`es`/`fa`/`pt`/`ar`/`id`) с
  автодетектом и командой `/language`, см. раздел «Мультиязычность» ниже.

## Первый запуск с LM Studio на хосте

Это основной режим: бот работает в Docker, а LM Studio запускается обычным локальным приложением на macOS.

1. Открой LM Studio.

2. В LM Studio открой `Developer` и включи `Start server`.

3. Загрузи любую подходящую chat/instruct модель. Бот берёт первую loaded-модель, если в `.env` стоит `LMSTUDIO_MODEL=auto`.

4. Подними бота:

```bash
docker compose up --build
```

5. Отправь боту `/start`, затем YouTube-ссылку.

Проверить, какие модели видит бот:

```text
/models
```

Если хочешь закрепить конкретную модель, укажи её id в `.env`:

```dotenv
LMSTUDIO_MODEL=publisher/model-id
```

Если хочешь, чтобы бот сам пытался загрузить указанную модель через LM Studio REST API:

```dotenv
LMSTUDIO_AUTO_LOAD=true
```

По умолчанию `LMSTUDIO_AUTO_LOAD=false`, чтобы случайно не загрузить тяжёлую модель в память.

Для моделей, загруженных с маленьким контекстом, например `4096`, бот режет transcript на небольшие части:

```dotenv
TRANSCRIPT_CHUNK_MAX_CHARS=3000
LLM_MAX_TOKENS=1200
```

Для Qwen3 8B с context length `16384` текущий профиль такой:

```dotenv
LMSTUDIO_NUM_CTX=16384
TRANSCRIPT_CHUNK_MAX_CHARS=10000
LLM_MAX_TOKENS=4000
```

`LMSTUDIO_NUM_CTX` используется при автозагрузке модели через LM Studio API. Если `LMSTUDIO_AUTO_LOAD=false`, context length нужно выставить в LM Studio UI перед загрузкой модели. Размер чанка выбирается с запасом: transcript занимает только часть окна, потому что в контекст ещё попадают system prompt, инструкция и будущий ответ модели.

## OpenRouter как альтернативный LLM-провайдер

Помимо LM Studio, бот умеет ходить в OpenRouter (`LLM_PROVIDER=openrouter`) —
платный или бесплатный chain моделей, с дневным бюджетом
(`OPENROUTER_DAILY_BUDGET_USD`, `OPENROUTER_DAILY_REQUEST_LIMIT`) и circuit
breaker'ом: после серии ошибок провайдер помечается недоступным и бот
перестаёт долбить его запросами до остывания тайм-аута. Владелец переключает
paid/free через `/llm_paid` и смотрит статус через `/llm_mode`.

## Хранилище: SQLite

Всё состояние бота живёт в одной SQLite-базе:

```text
data/bot.db
```

Путь настраивается через `DATABASE_PATH` (по умолчанию — `<BOT_DATA_DIR>/bot.db`,
то есть `/data/bot.db` внутри контейнера). В базе — пользователи, кэш саммари,
дайджесты и их закрепления, состояние мониторинга каналов, персистентная
очередь задач, бюджет и рантайм-состояние OpenRouter. Используется stdlib
`sqlite3` в режиме WAL с одним процесс-локальным соединением.

Если на диске остались старые JSON-файлы состояния (`users.json`,
`summary_cache.json`, `digests.json`, `digest_pins.json`,
`monitoring_state.json`, `openrouter_budget.json` и т.п.) — при первом запуске
после обновления бот автоматически импортирует их содержимое в SQLite и
переименовывает файл в `*.migrated` (например, `users.json` →
`users.json.migrated`). Файл остаётся на диске как бэкап и в дальнейшем не
читается. Если `*.migrated`-файлов нет — значит миграция уже прошла или база
создана с нуля, это нормально.

Персистентная очередь означает, что если бот перезапустится с ссылками в
очереди (рестарт контейнера, обновление образа), незавершённые задачи не
теряются — они восстанавливаются и продолжают обрабатываться после старта.

## Мультиязычность

Бот поддерживает 7 языков интерфейса и саммари: `ru`, `en`, `es`, `fa`, `pt`,
`ar`, `id`.

- Язык определяется автоматически по `language_code` из Telegram-профиля при
  первом обращении; неподдерживаемые коды (например `de`) падают на `en`
  (`DEFAULT_LANG`).
- Переключить язык вручную можно командой `/language` — ручной выбор
  перебивает автодетект.
- Резолюция языка: ручной выбор из БД → сохранённый ранее автодетект →
  `language_code` из Telegram → `en`.
- Все пользовательские строки (онбординг, статусы, ошибки, кнопки, тексты
  саммари) берутся из JSON-локалей `app/locales/{lang}.json` — плоский
  словарь `ключ → текст` с плейсхолдерами вида `{name}`. В SQLite хранится
  только выбор языка пользователя (таблица `user_langs`), переводы живут
  исключительно в файлах локалей.
- Правила перевода: плейсхолдеры и HTML-теги (`<b>`, `<i>`, `<code>`,
  `<a href=…>`) сохраняются буквально, `/команды` не переводятся, числа —
  западные цифры даже в fa/ar. Соответствие проверяется структурными тестами
  (`tests/test_i18n.py`): паритет ключей, плейсхолдеров и команд между всеми
  локалями.
- Краткое саммари и полный конспект в Telegra.ph генерируются на языке
  пользователя (языковая директива добавляется к системному промпту
  суммаризатора); канонизация тегов через общий каталог остаётся только для
  русского языка.
- Кэш саммари — по-язычный: один и тот же ролик, запрошенный на разных
  языках, генерируется и кэшируется отдельно для каждого языка (ключ кэша —
  `video_id` для `ru`, ради обратной совместимости со старым кэшем, и
  `video_id:lang` для остальных языков). Значит, что один ролик может занимать
  до 7 отдельных записей кэша — по одной на язык. `/cache_drop` и
  колбэк-очистка удаляют записи ролика сразу на всех языках.
- Язык запланированных задач мониторинга каналов и всех owner-поверхностей
  (`/stats`, `/users`, `/scan_now`, `/refund`, `/prompt_*`, `/llm_*`,
  `/cache_drop`, `/queue`, `/models`, `/model`, owner-ветка `/help`, утренний
  дайджест, уведомления мониторинга) — всегда русский, независимо от того,
  какой язык выбрал себе владелец бота как обычный пользователь.

## Мониторинг YouTube-каналов и утренний дайджест

Бот может сам отслеживать список YouTube-каналов и раз в сутки (в заданное
время и таймзоне) искать новые ролики, фильтровать их по правилам и
генерировать саммари в фоне — без участия пользователя в моменте.

Правила мониторинга лежат в `data/monitoring.yaml` (создаётся с шаблоном по
умолчанию при первом запуске мониторинга). Основные поля:

```yaml
channels: []                # список отслеживаемых каналов (channel_id, channel_url, channel_name)
shows:
  whitelist: []              # подстроки в title/description — фильтр "точно интересно"
  blacklist: []               # подстроки — фильтр "точно пропустить"
experts:
  whitelist: []                # имена гостей/экспертов, которых стоит ловить
  blacklist: []
interests: []                   # темы пользователя — используются LLM для ранжирования утреннего дайджеста
min_duration_sec: 300
scan_time: "22:00"
scan_tz: Europe/Moscow
expert_segment_threshold_sec: 2400
expert_window_pre_sec: 60
expert_window_post_sec: 180
expert_mention_cluster_gap_sec: 300
```

`interests` — свободный список тем/ключевых слов; вместе с `shows.whitelist`
и `experts.whitelist` он передаётся LLM при формировании утреннего дайджеста,
чтобы отранжировать найденные за сутки видео по релевантности.

Важно: мониторинг **не шлёт** саммари по каждому ролику отдельным сообщением.
Все саммари за скан складываются в очередь дайджеста, и когда пачка
scheduled-задач полностью обработана, бот один раз зовёт LLM отранжировать
видео по интересам и присылает **одно** сообщение со списком (оценка 0–10,
ссылка на Telegra.ph-конспект, короткое обоснование). Если LLM ранжирования
недоступна — дайджест всё равно уходит, но без оценок (fallback).

Владелец может запустить скан вручную (`/scan_now`) или прервать его
(`/scan_stop`); отправка ссылки на канал в чат добавляет его в `channels`
автоматически.

## Cookies для закрытых или age-restricted роликов

Если ролик требует авторизации, экспортируй cookies из браузера аккаунта, который имеет доступ к ролику, и положи файл сюда:

```text
data/youtube.cookies.txt
```

Файл не коммитится в git.

## Премьеры и запланированные стримы

Если прислать ссылку на ролик, который ещё не вышел (премьера или
запланированный стрим), бот запомнит время выхода и вернётся за саммари
через `PREMIERE_SUMMARY_DELAY_HOURS` часов после релиза (по умолчанию 4 —
чтобы у YouTube успели появиться субтитры). Отложенная задача хранится в
базе (`status=deferred`) и переживает рестарт контейнера; фоновый
deferred-scheduler проверяет созревшие задачи каждые 5 минут. В логах:
`job.premiere.deferred`, `deferred.requeued`.

## Монетизация (PUBLIC_MODE)

По умолчанию (`PUBLIC_MODE=false`) бот закрыт allowlist'ом, как и раньше —
пользователи из `ALLOWED_USER_IDS`/`/user_add` работают без лимитов. Ставить
`PUBLIC_MODE=true` в `.env`, чтобы открыть бота внешним пользователям с
квотами и Stars-подпиской.

**Модель доступа:**

- Пользователи из allowlist (владелец и добавленные через `/user_add`) —
  безлимит, как всегда.
- Новый внешний пользователь получает **3 стартовых саммари** (lifetime,
  `QUOTA_STARTER`).
- После исчерпания стартовых — **1 бесплатное саммари в неделю**
  (`QUOTA_FREE_WEEKLY`, скользящее окно 7 дней).
- Подписка — **149 ⭐ Telegram Stars в месяц** (`SUBSCRIPTION_PRICE_STARS`) за
  **30 саммари в месяц** (`QUOTA_SUB_MONTHLY`, скользящее окно 30 дней),
  автопродление.
- Ролики без субтитров длиннее `HEAVY_DURATION_SEC` (по умолчанию 1 час)
  требуют транскрибации через Groq и списывают **×2** лимита; бот
  предупреждает об этом перед постановкой в очередь.
- Повторный запрос ролика, который уже есть в кэше саммари, **не списывает
  лимит** — кэш-хиты всегда бесплатны.
- Списание происходит **после** успешной генерации и доставки саммари, а не
  при постановке в очередь — если генерация упала, лимит не тратится.
- Под каждым саммари — кнопка **«📄 Транскрипт (md)»**: скачать полный
  транскрипт ролика markdown-файлом (таймкоды + текст, удобно для Obsidian и
  т.п.). Доступна allowlist-пользователям и подписчикам; внешний
  бесплатный пользователь получает alert с `/subscribe` вместо файла.
  Файлы лежат в `data/transcripts/<video_id>.md` и пишутся при каждой
  генерации саммари — ролики, обработанные до появления этой функции,
  файла не имеют (бот сообщает об этом при нажатии кнопки).

**Команды:**

- `/limits` - показать пользователю остаток лимита и когда он обновится.
- `/subscribe` - оформить подписку: бот присылает нативный Telegram-инвойс
  на `SUBSCRIPTION_PRICE_STARS` ⭐ с автопродлением каждые 30 дней
  (`create_invoice_link` + `subscription_period`).
- `/paysupport` - контакт/инструкция по вопросам оплаты (обязательное
  требование Telegram для ботов с платежами).
- `/refund <user_id> [charge_id]` (только владелец) - вернуть платёж через
  `refund_star_payment`; если `charge_id` не указан, берётся последний
  известный charge_id пользователя.

### Маршрутизация LLM по тарифам

Каждый job получает свой `route` для `LLMClient.generate` в зависимости от
того, кто его запустил (см. `_process_youtube_job` в `app/pipeline.py`):

| Пользователь | `llm_route` | Поведение |
| --- | --- | --- |
| Allowlist (владелец, `/user_add`) | `default` | Обычный режим: paid/free зависит только от глобального тоггла `/llm_paid`. |
| Подписчик (`is_subscriber=True`) | `paid_fallback` | Сначала free-цепочка; если она медленнее `PAID_FALLBACK_FREE_BUDGET_SEC` (по умолчанию 180 сек), исчерпалась или circuit breaker открыт — автоматически уходит на платную модель OpenRouter. |
| Free-внешний пользователь | `free_only` | Только free-цепочка, всегда — платные токены на него никогда не тратятся, независимо от глобального `/llm_paid`. |

Это значит, что подписчики защищены от простоев/медленности free-цепочки
автоматически, без ручного переключения `/llm_paid` владельцем — платная
фолбэк-модель включается для них по требованию на уровне job'а, а не
глобально для всего бота.

**Логи маршрутизации:**

```text
job.llm_route job_id=... route=default|paid_fallback|free_only
llm.paid_fallback.trigger reason=breaker_open|slow|free_exhausted
llm.paid_fallback.success model=...
```

**Опционально, для скорости ответа:** можно всё равно включить `/llm_paid`
перед `PUBLIC_MODE=true` в проде, чтобы allowlist-пользователи и подписчики
сразу шли на платную модель без ожидания free-цепочки — но это больше не
обязательно: подписчики получают платный fallback автоматически при
проблемах с free-цепочкой, даже если глобальный `/llm_paid` выключен.

**Известное ограничение:** очередь генерации персистентна и переживает
рестарт бота (см. «Хранилище: SQLite» выше), но `quota_user_id` — кому
списывать лимит за конкретный job — не сохраняется в таблице `jobs` и
теряется при восстановлении очереди после рестарта. Если на момент рестарта
в очереди был job внешнего (не-allowlist) пользователя, после рестарта он
всё равно будет обработан и саммари доставлено, но лимит за него **не
спишется** (job восстанавливается с `quota_user_id=None`). Это редкий кейс
(рестарт ровно с внешним job'ом в очереди) и он принят осознанно — стоимость
починки (персистить и восстанавливать владельца квоты) не оправдана риском
редкой бесплатной генерации.

**Логи:** ключевые события биллинга в логах бота:

```text
billing.boot public_mode=... price_stars=... quotas=starter/free_weekly/sub_monthly
billing.subscription.activated user_id=... until_unix=... charge_id=...
billing.usage.charged user_id=... video_id=... weight=... kind=...
billing.charge_failed user_id=...
billing.payment.no_user_or_store payload=...
```

**Аналитика внешних пользователей:** события `first_start` (deep_link/organic),
`quota_denied`, `sub_activated`, `sub_renewed` пишутся в append-only таблицу
`analytics_events` и копятся бессрочно (лог-событие: `analytics.first_start`).
Воронка за последние 30 дней (`/start` → первая генерация → упёрлись в лимит →
подписка → продления) показывается владельцу в `/stats`.

**Ручной чеклист владельца после включения `PUBLIC_MODE=true`:** зайти с
второго (не-allowlist) аккаунта → 3 стартовых саммари проходят → 4-й запрос
упирается в лимит с кнопкой оформления подписки → `/subscribe` → оплатить
149 ⭐ реальными звёздами → саммари проходит → `/limits` показывает остаток →
`/refund` с owner-аккаунта возвращает платёж.

## Browser extension

В `browser-extension/` — расширение для Chrome/Firefox (Manifest V3),
добавляющее кнопку **🔮 Summary** на странице YouTube-видео рядом с
Like/Share/Save. Клик открывает Telegram deep-link'ом на бота с `video_id` —
бот ставит ролик в очередь и присылает саммари в личный чат. Хэндл бота
настраивается через options-страницу расширения. Подробности установки — в
`browser-extension/README.md`.

## Команды

Обычным пользователям доступны только:

- `/start` - краткая инструкция (либо, если передан deep-link с video_id — сразу ставит ролик в очередь).
- `/help` - что умеет бот.
- `/last` - последние 20 саммари пользователя.
- `/limits` - остаток лимитов (актуально при `PUBLIC_MODE=true`, см. раздел «Монетизация» ниже).
- `/language` - выбрать язык бота (7 языков; ручной выбор перебивает автодетект, см. раздел «Мультиязычность»).
- `/subscribe` - оформить подписку за Telegram Stars.
- `/paysupport` - вопросы по оплате.

Владельцу (`OWNER_USER_ID`) дополнительно доступны:

- `/refund` - вернуть платёж Stars пользователю (см. раздел «Монетизация»).
- `/users` - список пользователей с доступом.
- `/user_add 123456789 Имя` - добавить пользователя.
- `/user_remove 123456789` - удалить пользователя.
- `/models` - показать модели, которые видит локальный LLM-сервер.
- `/model` - показать модель, которую бот использует для summary.
- `/queue` - показать текущую очередь генерации summary.
- `/stop` или `stop` - остановить текущую генерацию summary и очистить очередь.
- `/scan_now` - запустить мониторинг сейчас (или показать статус идущего скана).
- `/scan_stop` - прервать запущенный мониторинговый скан.
- `/llm_mode` - показать активный LLM-провайдер и режим (free/paid).
- `/llm_paid` - переключить OpenRouter между paid и free (тоггл).
- `/stats` - статистика бота за 30 дней.
- `/cache_drop` - убрать ролик из кэша саммари (аргумент — video_id или URL, либо интерактивно); заодно удаляет сохранённый md-транскрипт этого ролика.
- `/prompt_set` - задать кастомный системный промпт для саммари (бот ждёт текст 5 минут).
- `/prompt_show` - показать текущий системный промпт.
- `/prompt_reset` - вернуть системный промпт к дефолту.

Если прислать несколько YouTube-ссылок подряд, бот поставит их в общую очередь и будет генерировать summary по одному ролику за раз, в порядке получения ссылок. Служебное сообщение текущей генерации обновляется и при добавлении новых ссылок переносится в конец чата. Если в очереди несколько ссылок, в служебном сообщении появляется строка вида `очередь: 1/3`.

Если генерация конкретного ролика завершилась ошибкой, бот пришлёт отдельное сообщение с гиперссылкой на видео и причиной ошибки, а очередь перейдёт к следующей ссылке.

## Логи

Смотреть текущий прогон:

```bash
docker compose logs -f bot
```

Последние строки:

```bash
docker compose logs --tail=200 bot
```

Быстро посмотреть последние ошибки сразу из файла и из `docker logs`:

```bash
./scripts/log-errors.sh
```

Можно передать число строк, например:

```bash
./scripts/log-errors.sh 300
```

Файл-лог бота сохраняется в:

```text
data/logs/bot.log
```

Этот файл ротируется раз в сутки, хранится до 30 архивов. Отдельно Docker-логи контейнера ограничены по размеру и количеству файлов через `docker-compose`, чтобы stdout/stderr тоже не разрастались бесконечно.

При старте бот пишет в лог факт открытия базы и (если были легаси-JSON) их миграцию, а восстановление очереди после рестарта видно отдельной строкой:

```text
db.boot path=/data/bot.db
users.migrated count=...
summary_cache.migrated entries=...
monitoring.state.migrated rows=...
queue.restored jobs=...
```

`*.migrated` строки появляются только при первом запуске после обновления, если на диске оставались старые JSON-файлы состояния — их отсутствие не ошибка, а признак того, что база уже мигрирована или создана с нуля.

В логах есть `job_id`, по которому удобно отслеживать один ролик от старта до конца:

```text
job.start
queue.job.enqueued / queue.job.start / queue.job.done / queue.job.failed / queue.job.cancelled
job.metadata.done
job.transcript.done source=youtube
job.transcript.unavailable fallback=groq_queue
transcription_queue.audio_download.done
groq.transcribe.done
job.chunking.done
summary.chunk.start / summary.chunk.done
summary.synthesis.start
summary.done
telegraph.publish.done
job.done
job.failed
```

Утренний дайджест мониторинга оставляет свои события:

```text
morning_digest.ranked items=... ranked=...
morning_digest.rank_failed
morning_digest.sent items=... chat_id=...
```

`morning_digest.rank_failed` — не фатальная ошибка: если LLM-ранжирование недоступно, дайджест всё равно уходит, просто без оценок.

Если LM Studio не вернула ответ на один LLM-вызов за 20 минут, бот делает один retry этого же вызова после короткой паузы. В логах это видно как:

```text
llm.generate.timeout_retry
llm.generate.retry_success
```

## Ограничения MVP

- Первый запуск локальной LLM может быть медленным из-за загрузки модели.
- Telegra.ph-токен можно оставить пустым: бот создаст временный аккаунт при запуске.
