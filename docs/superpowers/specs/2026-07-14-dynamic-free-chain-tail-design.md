# Динамический хвост free-цепочки OpenRouter

Дата: 2026-07-14. Статус: одобрено владельцем.

## Проблема

Free-цепочка (`OPENROUTER_MODEL_FREE_CHAIN`) — статический список. Когда все
её модели отказывают (перегрузка 429 у всех провайдеров разом, массовое
исчезновение free-тарифов, как у `openai/gpt-oss-120b:free` 2026-07-13),
job падает, хотя в каталоге OpenRouter могут оставаться живые бесплатные
модели. Событие редкое (0 исчерпаний за июль 2026), фича — страховка.

## Решение

После полного исчерпания сконфигурированной цепочки (все модели, все проходы)
`OpenRouterClient._generate_free_chain` перед выбросом ошибки пробует
«динамический хвост» — до 3 free-моделей из актуального каталога.

### Отбор моделей хвоста

1. `GET {base_url}/models`, кэш ответа в памяти клиента с TTL 1 час
   (защита от долбёжки каталога во время шторма 429).
2. Кандидаты: id с суффиксом `:free`, которых нет в сконфигурированной
   цепочке (сравнение по полному id).
3. Фильтр непригодных для универсальных русских саммари: отбрасываем id,
   содержащие подстроки `coder`, `code`, `safety`, `vl`, `vision`, `guard`
   (сравнение case-insensitive по id без вендорного префикса), и модели
   с `context_length < 131072` (промпты бота — до ~30k токенов).
4. Сортировка по убыванию `context_length`, максимум 3 модели.

На каталоге 2026-07-14 хвост при текущей цепочке:
`poolside/laguna-xs-2.1:free`, `poolside/laguna-m.1:free`,
`google/gemma-4-31b-it:free`.

### Проход по хвосту

- Один проход (без повторных pass'ов с паузами — это глубокий резерв).
- Та же обработка, что у основной цепочки: `_generate_one_attempt` со всеми
  её механизмами (reasoning exclude, ретраябельные 429/402/404/5xx,
  `_OpenRouterTruncated` → следующая модель, обрезанный текст как
  last resort вместе с основной цепочкой).
- Успех хвоста НЕ трогает circuit breaker (`record_success`) — здоровье
  сконфигурированной цепочки он не отражает. Провал хвоста дополнительный
  `record_failure` не добавляет (failure уже записан при исчерпании цепочки).

### Отказоустойчивость

- Ошибка `GET /models` (сеть, 5xx, кривой JSON) → хвост молча пропускается,
  поведение идентично текущему. Логируется warning.
- Пустой хвост после фильтра → то же самое.

### Наблюдаемость

- `llm.generate.dynamic_tail.try model=<id>` на каждую попытку;
- `llm.generate.dynamic_tail.success model=<id>`;
- `llm.generate.dynamic_tail.skipped reason=catalog_error|empty` при пропуске.

Итоговое сообщение об исчерпании (когда и хвост не помог) упоминает,
сколько моделей хвоста было испробовано.

## Не делаем (YAGNI)

- Никакого персиста каталога на диск — только in-memory TTL.
- Никакой оценки качества моделей — только фильтр по id и контексту.
- Динамический хвост не участвует в обычной работе цепочки и не
  переупорядочивает сконфигурированные модели.

## Тесты (tests/test_dynamic_tail.py, стиль test_truncated_output.py)

1. Цепочка исчерпана (все 429) → хвост подхватывает, результат возвращается,
   лог success.
2. Фильтр: coder/safety/vl/vision/маленький контекст отсеяны; лимит 3;
   модели из сконфигурированной цепочки исключены; сортировка по контексту.
3. `GET /models` падает → прежняя ошибка «цепочка исчерпана», без новых исключений.
4. Каталог кэшируется: два исчерпания подряд в пределах TTL → один запрос каталога.
5. Хвост тоже отказал → ошибка исчерпания, breaker не получает лишний failure.
