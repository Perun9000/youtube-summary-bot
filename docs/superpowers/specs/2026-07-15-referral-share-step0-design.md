# Реферальный шеринг саммари — ступень 0 (RAT)

Дата: 2026-07-15. Статус: одобрено владельцем (обсуждение 2026-07-15).

## Гипотеза и что проверяем

MGM-механика: пользователи шерят саммари в чаты, читатели приходят по
реф-ссылке, часть конвертируется в пользователей и подписчиков. Ступень 0 —
самый дешёвый тест самого рискованного допущения: **конвертит ли шер в
клики, новых пользователей и оплаты**. Никаких наград на этой ступени.
Кнопка шеринга видна ТОЛЬКО владельцу (тихий тест на своих чатиках).

## Механика

### 1. Кнопка «Поделиться» под саммари (только владелец)

- В `_build_summary_keyboard` (app/delivery.py) добавляется ряд с кнопкой
  `📤 Поделиться` (`callback_data="share:<video_id>"`), только когда
  `chat_id == settings.owner_user_id`.
- Обработчик callback'а (app/bot_handlers.py) берёт саммари из
  `summary_cache.get_any(video_id)` и присылает владельцу **отдельное
  шер-сообщение**, готовое к форварду в любой чат. Callback отвечает
  `answer()` без alert'а. Если кэша нет (протух) — ответ-alert
  «Саммари уже не в кэше — перегенерируй ролик».
- Аналитика: `analytics.record(user_id, "share_button", video_id)`.

### 2. Шер-сообщение (то, что форвардится в чатики)

Компактное, полезное само по себе, но не исчерпывающее:

```
🎬 <b>{title}</b>
{channel_name}

{2–3 первых предложения overview}

⏱ Чтение полного саммари: {read_time} мин
🔮 Полное саммари: {ref_link}

через @YouTube_Sum_mary_bot
```

- `ref_link` = `https://t.me/<bot_username>?start=r{owner_id}_{video_id}`
  (payload ≤ 64 символов: `r` + uid до 10 цифр + `_` + video_id 11 симв.).
- Без inline-кнопок, ссылка — текстом/анкором в HTML. Причины: (1) анкор
  переживает и форвард, и копипаст, кнопка — только форвард; (2) сообщение
  с кнопкой выглядит как бот-реклама, а сила шера — в виде личной
  рекомендации; (3) по метрикам разницы нет — клик и по кнопке, и по анкору
  наблюдаем одинаково, только через входящий /start.
- Первые предложения overview берутся до ~300 символов (обрезка по границе
  предложения), HTML-эскейп как в остальной доставке.

### 3. Приём реф-ссылки: `/start r<uid>_<video_id>`

В обработчике `/start` (app/bot_handlers.py), ПЕРЕД существующей веткой
голого `<video_id>`:

- Regex payload'а: `^r(\d{1,12})_([A-Za-z0-9_-]{11})$`.
- Привязка реферала: только если это ПЕРВЫЙ контакт пользователя с ботом
  вообще (`record_first_start(user_id, "referral")` вернул true — существующие
  пользователи ничьими рефералами не становятся) и `uid != user_id` —
  записать в новую таблицу
  `referrals(user_id INTEGER PRIMARY KEY, referrer_id INTEGER NOT NULL,
  video_id TEXT NOT NULL DEFAULT '', created_at REAL NOT NULL)`
  через `INSERT OR IGNORE`. Повторный `/start` по любой реф-ссылке привязку
  НЕ меняет (first-touch: первый реферер выигрывает навсегда — иначе на
  ступени 1 появится стимул «перехватывать» чужих рефералов перед оплатой).
  Self-referral просто не пишется.
- Аналитика: `record_first_start(user_id, "referral")` для новых;
  `analytics.record(user_id, "ref_start", f"{referrer_id}:{video_id}")`
  для всех переходов.
- Выдача ценности: `summary_cache.get(video_id, lang)` → есть кэш на языке
  пользователя (или `get_any` фолбэком) — сразу отправить полное саммари
  стандартной доставкой (формат как у обычной выдачи, с Telegra.ph).
  Кэша нет — поставить видео в обычную очередь от имени этого пользователя
  (его free-квота; для нового пользователя это первая из 3/день —
  фактически бесплатно для него).
- После выдачи — одна строка питча (i18n-ключ `ref.pitch`):
  «Пришли мне ссылку на любое YouTube-видео — сделаю такое же саммари.
  {N} бесплатных в день.»
- PUBLIC_MODE-гейты (allowlist в приватном режиме) применяются как обычно —
  реф-ссылка не даёт обхода.

### 4. Метрики ступени 0 (смотрим руками через SQL/логи)

- `share_button` — сколько раз жали шер (и по каким видео).
- `ref_start` — клики по реф-ссылкам; `first_start source=referral` — новые.
- `referrals` — таблица привязок (фундамент для наград ступени 1).
- Оплаты рефералов: `SELECT ... FROM usage_events JOIN referrals ...` —
  без кода, руками при разборе итогов теста.

## Не делаем на ступени 0 (YAGNI)

- Никаких наград, балансов, начислений — только учёт привязок.
- Никакого inline mode / switch_inline_query — форвард руками.
- Кнопка не показывается никому, кроме владельца.
- Локализация шер-сообщения — только ru (тест на русскоязычных чатиках).

## Тесты

1. `_build_summary_keyboard`: владельцу — ряд с «Поделиться», не-владельцу — нет.
2. Парсер payload: `r123_abcDEF12345` → (123, video_id); мусор → мимо ветки;
   голый video_id работает как раньше (регресс browser-extension).
3. Привязка: новый пользователь пишется в `referrals`; повторный старт не
   перезаписывает; self-referral не пишется.
4. Формирование шер-текста: обрезка overview по предложению, HTML-эскейп,
   корректный ref_link.
5. `/start` по реф-ссылке с кэшем — доставка без постановки в очередь;
   без кэша — enqueue.
